Start the Admin Dashboard Module now.

Build the Admin Dashboard carefully and incrementally inside one module.
Do not rewrite existing working modules.
Do not break:

* /news
* /news/:id

* /articles/:id
* /articles/new

==================================================
A. ADMIN DASHBOARD PURPOSE
==========================

Create a complete management dashboard with role-based access.

Roles:

1. SUPER_ADMIN
   Full application management:

* Users/members management
* Roles management
* Articles management
* News management
* Broadcast management
* App overview/statistics
* Developer/app information visibility if useful
* Future settings placeholders if needed

2. ADMIN
   Can manage:

* Members/users, but with restrictions
* Articles
* News
* Broadcast

ADMIN must NOT:

* Promote anyone to SUPER_ADMIN
* Modify SUPER_ADMIN accounts
* Delete or deactivate SUPER_ADMIN accounts
* Remove SUPER_ADMIN role
* Change isDeveloper

3. MODERATOR
   Can manage only:

* News
* Articles

MODERATOR must NOT manage:

* Users/members
* Roles
* Broadcast
* SUPER_ADMIN/ADMIN settings

4. MEMBER
   Cannot access Admin Dashboard at all.

E. USERS MANAGEMENT — /admin/users
==================================

Access:

* SUPER_ADMIN
* ADMIN

MODERATOR and MEMBER forbidden.

Features:

1. List users
   Show:

* fullName
* account
* email
* role
* isActive/status
* professionGroup
* createdAt

2. Search/filter
   At minimum:

* Search by name/account/email if easy
* Filter by role if easy

3. User detail/edit panel or modal
   Allow editing:

* fullName
* account
* email
* phone
* whatsapp
* address
* birthDate
* professionGroup
* specialtyText
* isActive/status where allowed
* role where allowed

4. Role management rules

SUPER_ADMIN can:

* Change MEMBER ↔ MODERATOR ↔ ADMIN
* Create/assign ADMIN
* Create/assign MODERATOR
* Deactivate/reactivate users
* Edit normal users
* Edit ADMIN/MODERATOR/MEMBER
* View SUPER_ADMIN accounts

SUPER_ADMIN must be protected:

* Prevent removing the last SUPER_ADMIN.
* Prevent deactivating the last active SUPER_ADMIN.
* Prevent accidental self-demotion unless explicitly guarded. Prefer disallow self-demotion.

ADMIN can:

* Change MEMBER ↔ MODERATOR
* Deactivate/reactivate MEMBER and MODERATOR
* Edit MEMBER and MODERATOR profiles
* View users

ADMIN cannot:

* Promote anyone to ADMIN
* Promote anyone to SUPER_ADMIN
* Edit SUPER_ADMIN
* Edit another ADMIN role/status
* Deactivate ADMIN
* Deactivate SUPER_ADMIN
* Change isDeveloper

MODERATOR cannot access user management.

Backend must enforce all of these rules.
Frontend hiding is not enough.

Required API:

* GET /api/admin/users
* GET /api/admin/users/:id
* PATCH /api/admin/users/:id
* Optional: PATCH /api/admin/users/:id/role
* Optional: PATCH /api/admin/users/:id/status

Use whichever structure is clean, but permissions must be correct.

F. ARTICLES MANAGEMENT — /admin/articles
========================================

Access:

* MODERATOR
* ADMIN
* SUPER_ADMIN

Purpose:
Review and manage submitted articles.

Show list with filters:

* PENDING
* APPROVED
* REJECTED
* ARCHIVED
* All

Each article card/row:

* title
* authorName
* category
* status
* createdAt
* summary

Actions:
For PENDING:

* View
* Approve
* Reject with reason

For APPROVED:

* View
* Archive

For REJECTED:

* View
* Show rejection reason
* Optional approve again if safe

For DRAFT:

* Usually author-owned only; admin can view if needed, but do not publish drafts unless policy says so.

Required behavior:

* Approving sets status APPROVED, reviewedById, publishedAt.
* Rejecting sets status REJECTED, reviewedById, rejectionReason.
* Archiving sets status ARCHIVED.
* Approved articles appear in public /articles.
* Pending/rejected/archived do not appear publicly.

Use existing article backend endpoints if already available:

* approve/reject endpoints exist backend-only from Articles Module.
  If missing admin list endpoints, add them.

Required API if needed:

* GET /api/admin/articles
* GET /api/admin/articles/:id
* POST /api/admin/articles/:id/approve
* POST /api/admin/articles/:id/reject
* POST /api/admin/articles/:id/archive

Do not duplicate article logic if existing endpoints already work.

L. PERMISSION ENFORCEMENT
=========================

Do not rely only on frontend.

Backend must enforce:

* MEMBER cannot access admin APIs.
* MODERATOR cannot access user management.
* MODERATOR cannot manage broadcast.
* ADMIN cannot modify SUPER_ADMIN.
* ADMIN cannot promote to ADMIN or SUPER_ADMIN.
* ADMIN cannot edit/deactivate ADMIN or SUPER_ADMIN.
