==================================================
A. ARTICLES MODULE PURPOSE
==========================

Articles are different from News.

News:

* Official SGMA announcements and activities.

Articles:

* Scientific/professional content written by members or staff.
* Articles should be submitted by users.
* Articles should not become public automatically unless approved/published according to the status system.

For this phase, build:

1. Public approved articles reading experience.
2. Article detail page.
3. Write article page.
4. My Articles page.
5. Backend article APIs.
6. Status workflow foundation.

Do NOT build the full admin review dashboard now.
Admin review/approval UI will be implemented later in the Admin Dashboard phase.

D. ARTICLE STATUS WORKFLOW
==========================

For this phase:

  or published articles according to existing schema.

2. /articles/:id opens only:

* approved/public articles for general users
* author can open own article even if DRAFT/PENDING/REJECTED from My Articles if implemented cleanly.

   PENDING

Reason:

6. ARCHIVED:
   Not public.

==================================================
E. PERMISSIONS
==============

All authenticated users can:

* Read approved articles.
* Open approved article detail.
* Submit a new article.
* View their own articles in /articles/my.

Author can:

* See own DRAFT/PENDING/REJECTED/APPROVED articles.
* Edit own article only if status is DRAFT or PENDING.
* Delete/archive own article only if status is DRAFT or PENDING, if implemented.

MEMBER cannot:

* Approve articles.
* Publish articles.
* Reject articles.
* Edit other users’ articles.

MODERATOR / ADMIN / SUPER_ADMIN:
For this phase, may have backend-only approval endpoints if already simple, but do not build the full approval UI now.

If adding backend-only review endpoints, enforce:

* MODERATOR / ADMIN / SUPER_ADMIN can approve/reject/archive.
* MEMBER forbidden.

==================================================
F. BACKEND API REQUIREMENTS
===========================

Create clean JWT-protected APIs.

Public approved articles:

GET /api/articles

* Returns approved articles only.
* Sorted newest first.

GET /api/articles/:id

* Returns approved article.
* Also allow author to access own non-approved article if needed from My Articles.
* Return 404 if not found or not allowed.

My articles:

GET /api/articles/my

* Returns articles authored by current user.
* Includes all statuses.
* Sorted newest first.

Create article:

POST /api/articles

* JWT protected.
* Creates article for authenticated user.
* Never trust authorId from frontend.
* authorId must come from JWT.
* Requires title, summary, content, category if used.
* imageUrl optional.
* Default status = PENDING unless draft support is implemented.

Update own article:

PATCH /api/articles/:id

* Author can edit only own article.
* Only allowed if status is DRAFT or PENDING.
* Cannot edit APPROVED/REJECTED/ARCHIVED unless staff rules are added.
* Must not allow authorId/status/reviewedById injection from frontend.

Delete/archive own article:

DELETE /api/articles/:id

* Soft delete/archive preferred.
* Author can archive only own DRAFT/PENDING.
* Staff can archive later if backend review endpoints exist.

Optional staff review endpoints:
POST /api/articles/:id/approve
POST /api/articles/:id/reject
Only MODERATOR / ADMIN / SUPER_ADMIN.
No frontend admin UI required now.
