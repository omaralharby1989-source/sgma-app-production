A. REQUIRED BEHAVIOR CHANGE
===========================

Task creators must be able to select any ACTIVE user as part of a task:

Allowed task participants:

* MEMBER
* MODERATOR
* ADMIN
* SUPER_ADMIN

Do NOT restrict task assignees to MEMBER only.

Why:
Sometimes a manager, admin, moderator, or super admin may be responsible for a task or supporting it.

==================================================
B. TASK PARTICIPANT ROLES
=========================

Add role distinction inside each task.

Each task should support:

1. مشرف المهمة
   field:
   supervisorUserId

Rules:

* One user only.
* Can be MEMBER / MODERATOR / ADMIN / SUPER_ADMIN, as long as active.
* This person is the main responsible supervisor for the task.

2. المكلفون
   field/table relation:
   assignees

Rules:

* Multiple users allowed.
* Any ACTIVE user role allowed.
* These are the main people assigned to complete the task.

3. المساعدون / الداعمون
   field/table relation:
   supporters

Rules:

* Multiple users allowed.
* Any ACTIVE user role allowed.
* These are support/helper people.

Important:
A user can be in one or more roles if needed, but avoid duplicates in the same role list.
If a user is selected multiple times, store each role cleanly or deduplicate safely.

Recommended DB design:
Either extend task_assignees with participantRole:

* ASSIGNEE
* SUPERVISOR
* SUPPORTER

OR create separate fields/tables if cleaner.

Preferred:
Use task_assignees with participantRole because it is flexible.

==================================================
C. ACCESS RULES
===============

A user can see a task if:

1. User is MODERATOR / ADMIN / SUPER_ADMIN
   OR
2. User is included in the task participants as:

   * SUPERVISOR
   * ASSIGNEE
   * SUPPORTER

A normal MEMBER who is not part of the task cannot view it.

A staff user who is not listed can still view all tasks because staff have full task management permissions.

Backend enforcement is mandatory.

==================================================
D. REPORTING RULES
==================

Any participant of a task can submit a task report:

* supervisor
* assignee
* supporter

Staff users can submit administrative notes/reports on any task.

Report type logic:

* If user is staff and is adding an administrative note from staff UI: ADMIN_NOTE
* If user is a participant adding a normal progress report from task detail: MEMBER_REPORT or PARTICIPANT_REPORT if you add that enum

Avoid the previous mistake:
Do not let the frontend decide report type in a way that can be spoofed.
Backend must determine report type based on route/context/user role.

Simple acceptable behavior:

* Reports submitted from /tasks/:id by a participant are MEMBER_REPORT.
* Reports submitted from admin/staff controls are ADMIN_NOTE.
