A. NEWS VIEW COUNT
==================

Add view count to news.

Behavior:

* Every time an authenticated user opens a news detail page, increment the view count.
* Avoid extreme duplicate counting on fast refresh if easy:

  * either count one view per user per news per day
  * or count once per session/localStorage
  * if too complex, simple increment is acceptable for first version, but document it.

Display:

* On /news list cards:

  * show عدد المشاهدات
  * icon: eye if available
* On /news/:id detail:

  * show عدد المشاهدات clearly near date/category.

Backend:

* Add view tracking table or viewCount column.
  Preferred:

  * add newsViews table with userId/newsId/createdAt for uniqueness
  * add denormalized viewCount if useful
    Acceptable simple version:
  * add viewCount integer to news table

API:

* GET /api/news/:id should register a view for authenticated users.
* Or add POST /api/news/:id/view if cleaner.

Security:

* userId from JWT only.
* Do not trust client userId.

C. REACTIONS FOR NEWS AND ARTICLES
==================================

Add Facebook-like reactions.

Reaction types:

* LIKE: أعجبني
* LOVE: أحببته
* SUPPORT: داعم
* THANKS: شكراً
* INSIGHTFUL: مفيد

Use icons if available:

* thumbs up
* heart
* hand/medical/support icon if available
* thanks icon
* lightbulb

Rules:

* Each authenticated user can have one active reaction per news item.
* Clicking a different reaction changes their reaction.
* Clicking the same reaction again removes it.
* Show total reaction count and reaction breakdown.
* Show the current user’s selected reaction highlighted.

Apply to:

* News detail page
* News cards if not too crowded: show summary only
* Article detail page
* Article cards if not too crowded: show summary only

Backend tables:

1. news_reactions

* id
* newsId
* userId
* reactionType
* createdAt
* updatedAt
  Unique:
* newsId + userId

2. article_reactions

* id
* articleId
* userId
* reactionType
* createdAt
* updatedAt
  Unique:
* articleId + userId

APIs:

  * viewCount

  * myReaction
* POST /api/news/:id/reaction
  body: { reactionType }
  sets/changes/removes reaction
  If reactionType is null or same reaction toggled, remove.

Articles:

* GET /api/articles/:id returns:

  * viewCount
  * reactionSummary
  * myReaction
* POST /api/articles/:id/reaction

Validation:

* reject invalid reactionType with 400.

D. CHAT ONLINE USERS
====================

Add online users/active users indicator in chat.

Goal:
Inside chat pages, show:

* عدد المتواجدين الآن
* list of names currently active in the chat

Applies to:

* /chat/public
* /chat/admin if meaningful
* /chat center if useful

Recommended first implementation:
Polling presence, not WebSocket.

Create table:
chat_presence

Fields:

* id
* userId
* roomType
* roomKey
* lastSeenAt
* createdAt
* updatedAt

roomType values:

* PUBLIC_CHAT
* ADMIN_DIRECT_CHAT

roomKey:

* PUBLIC for public chat
* for admin direct chat:

  * member user id or conversation key if needed

Behavior:

* When user opens /chat/public, frontend periodically calls:
  POST /api/chat/presence
  with roomType=PUBLIC_CHAT, roomKey=PUBLIC
* Server updates lastSeenAt for current user.
* GET /api/chat/presence?roomType=PUBLIC_CHAT&roomKey=PUBLIC returns users with lastSeenAt within last 60 seconds.
* Poll every 20–30 seconds.
* Online cutoff: 60–90 seconds.

Display:

* In public chat:

  * small card/sidebar/top drawer:
    المتواجدون الآن: X
  * list of names:
    fullName + role badge
* On mobile:

  * do not take too much space
  * use collapsible section:
    المتواجدون الآن
  * tap expands/collapses list.

For admin direct chat:

* If easy, show active users in the selected conversation.
* If not, apply only to public chat first and document admin direct chat as not implemented yet.
  But preferred:

  * ADMIN_DIRECT_CHAT roomKey = conversationUserId
  * member and staff in that conversation update presence there.

Security:

* userId from JWT.
* Only users allowed to access that chat room can update/read presence.
* Syria Academy-only users should not access chat at all, consistent with existing academy rules.
